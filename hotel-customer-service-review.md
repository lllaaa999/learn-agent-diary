# 酒店智能客服 Agent — 项目复盘

## 项目背景

2026年6月，在学习 AI Agent 应用开发的过程中，选择了"酒店智能客服"作为实战项目。目标是让 AI 代替人工回答客人关于房型、价格、入住退房、周边设施等常见问题。

技术选型：Dify Chatflow（低代码 AI 工作流平台），后端对接 DeepSeek 大模型，前端通过 API 提供对话服务。

## 业务场景

酒店客人常见问题分为几类：

| 类型 | 典型问题 |
|------|---------|
| 房型咨询 | "大床房和双床房有什么区别" "有没有带浴缸的房间" |
| 价格查询 | "标准间多少钱一晚" "周末价格一样吗" |
| 入住退房 | "几点可以入住" "可以延迟退房吗" |
| 周边服务 | "附近有地铁站吗" "酒店有停车场吗" |
| 投诉建议 | "房间太吵了" "空调不制冷" |

## Prompt 迭代过程

### 第一版：简单角色设定

直接告诉 AI "你是一个酒店客服"，结果问题很多：AI 会假装自己是人类、编造不存在的服务、在不知道答案时胡编乱造。

### 第二版：添加约束规则

加了5条硬规则：
1. **禁止角色扮演**：不能说"我是前台小王"，只能说"根据酒店信息"
2. **不确定就说不确定**：没有相关信息时回答"抱歉，我暂时无法确认"
3. **不承诺做不到的事**：比如客人要求免费升级，不能直接答应
4. **引导到人工**：涉及投诉、退款等敏感问题，引导联系前台
5. **不使用推理标签**：DeepSeek 返回的 `<think>` 标签要过滤掉

### 第三版：加入知识库

把酒店 FAQ 文档上传到 Dify 知识库，让 AI 基于真实资料回答。这样房型、价格、政策都有据可查，大幅减少幻觉。

## 遇到的问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| AI 角色扮演 | Prompt 约束不够 | 加"禁止角色扮演"硬规则 |
| 自循环回复 | AI 把自己当用户继续提问 | 限制单次回复长度，不允许反问 |
| `<think>` 标签泄露 | DeepSeek 模型特性 | 输出前过滤 `<｜end▁of▁thinking｜><｜end▁of▁thinking｜>AI 回复过长 | 知识库检索结果太多 | 限制检索条数为3条 |
| 中文乱码 | 编码问题 | 统一 UTF-8，设置 `charset=utf-8` |

## 最终效果

- 常见问题（房型/价格/政策）回答准确率 >90%
- 未知问题能诚实说"不确定"，不再胡编
- 投诉类问题自动引导到人工客服
- 单次回复控制在 200 字以内，简洁明了

## 经验总结

1. **Prompt 比模型重要**：同样的 DeepSeek 模型，加不加约束规则效果天差地别
2. **知识库是核心**：没有知识库的客服只是在"假装知道"，有了知识库才能"真知道"
3. **先限制再放开**：一开始把规则写死，跑稳了再逐步放宽
4. **Dify 适合快速验证**：从想法到能跑只用半小时，但深度定制需要代码能力
